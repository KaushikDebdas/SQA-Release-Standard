05.
Category 5 of 25 · one-time password

OTP is a second factor, not a placeholder.

A weak OTP flow defeats the entire purpose of two-factor authentication. Get the basics right or do not ship it.

Bad — Will be rejected
Length: 4 digits — sometimes 5, sometimes 6
GET /api/verify?otp=1234&user=42
// in URL — leaks to logs, browser history, referrer
server.log: "OTP for user 42 = 1234"
// plaintext in logs
No expiry. No attempt limit. Reusable.
  • OTP length is inconsistent — sometimes 4, sometimes 5, sometimes 6
  • OTP sent in URL query string — appears in logs, browser history, referer headers
  • OTP logged in plaintext on the server
  • No expiry — same code accepted hours later
  • Reusable — same code can verify multiple times
  • No rate limit — brute force succeeds in minutes
  • No "Resend" cooldown — endpoint can be spammed
Good — Passes review
Auto-detected from SMS
Code expires in 02:14 · Resend (4 left)
POST /api/verify { "otp": "[redacted]" }
// HTTPS, body-only, never logged
  • OTP length is always 6 digits — never less, never more
  • OTP transmitted in HTTPS POST body — never in URL or query string
  • Server logs only request ID and success/failure, never the OTP itself
  • OTP expires after 5 minutes; expired codes always rejected
  • Single-use — second attempt with the same code returns "already used"
  • After 5 failed attempts, OTP is invalidated and user must request a new one
  • Mobile input uses autocomplete="one-time-code" for SMS auto-fill
  • Resend has a cooldown timer; max 5 resends per hour
  • OTP is cryptographically random, not predictable patterns

07.
Category 7 of 25 · forgot password flow

Reset is a security boundary, not a form.

A password reset link is an authentication bypass if it leaks. Treat it like one.

Bad — Will be rejected
User entered: notreal@example.com → "No account found. Please register."
Reset link: http://app.example.com/reset?token=abc123 HTTP, never expires, reusable.
  • Reset link uses http:// — not HTTPS — so token can be intercepted
  • Token has no expiry; valid forever once issued
  • Token is reusable — clicking the link twice still works
  • Error message reveals whether an email is registered (account enumeration)
  • Old password keeps working after reset; sessions on other devices stay alive
  • No CAPTCHA / rate limit — endpoint can be abused to spam users with reset emails
Good — Passes review
If an account exists for this email, a reset link has been sent. Please check your inbox.
Reset link: https://app.example.com/reset?token=… HTTPS, single-use, expires in 30 min.
  • Reset link uses https:// only
  • Token is single-use and expires within 15–30 minutes
  • Same neutral message for valid + invalid emails — no enumeration
  • On successful reset, all active sessions on other devices are logged out
  • Old password is rejected immediately after reset
  • CAPTCHA and IP rate limit applied to prevent reset-email spam
  • Email arrives in inbox (DKIM/SPF/DMARC configured), not spam folder

Category 9 of 25 · destructive actions

Delete cannot be a single click.

Every "Are you sure?" you skip is a support ticket waiting to happen — sometimes a lawsuit.

Bad — Will be rejected
Single click → record gone. No confirm. No undo. No log.

// hard delete · no audit log · no permission check
Children rows orphaned in the database. Foreign keys broken.
  • Single click deletes immediately — no confirmation
  • No undo, no recovery — destructive and silent
  • Hard delete; no soft-delete flag, no audit log
  • No RBAC check — any user can delete any record
  • Cascade rules undefined — child rows orphan in the database. Example
  • No success message — user is not sure if anything happened
  • Bulk delete deletes everything when nothing is selected
  • Delete keyboard shortcut wipes records without warning
Good — Passes review

Toast: Deleted. Undo · 8s

// soft delete + audit log + RBAC enforced
  • Confirmation modal names the record and the consequences explicitly
  • Cancel + Confirm clearly distinguished; destructive action is the non-default
  • Esc dismisses, Tab cycles between buttons; modal is keyboard-accessible
  • Soft delete by default; restorable within a documented window (e.g. 30 days)
  • Undo toast appears for ~8 seconds after deletion
  • Bulk delete confirmation states the exact count: "Delete 5 records?"
  • RBAC enforced server-side — direct API calls without permission return 403
  • Audit log records user, timestamp, IP, record ID
  • Cascade behavior matches business rules — block, set null, or cascade — never silent orphans
  • Touch-friendly on mobile; modal readable without scrolling

21.
Category 21 of 25 · links

Links are the product's skeleton.

Every broken link is a dead end. Every "click here" is a missed accessibility signal. Every external link without rel is a security hole.

Bad — Will be rejected
For the documentation, click here.
For pricing details, click here.
For partner integration, click here.
External: http://partner-site.com
All "click here". HTTP, no rel="noopener". 404s in production.
  • Generic "click here" / "more" text — no context, no accessibility
  • Link text indistinguishable from body text — same color, no underline
  • External links use http:// — credentials and tokens leak in transit
  • External links open in new tab without rel="noopener noreferrer" — reverse-tabnabbing risk. Example
  • Multiple 404 / dead links present in production
  • Anchor (#section) links do not scroll smoothly; target hidden behind sticky header
  • mailto: and tel: links not used; phone numbers as plain text
  • No focus indicator on Tab — keyboard users lose place
Good — Passes review
Read the Middleware integration guide.
Compare enterprise pricing tiers.
For partner setup, see Acme Routing setup checklist.
External: partner-site.com (opens in new tab · noopener)
Email: support@erainfotech.com
Descriptive text · HTTPS · noopener · checked monthly for 404s
  • Link text describes the destination — never "click here", never the bare URL
  • Visually distinct — colored, underlined; hover and visited states differ
  • External links use https:// only
  • External target="_blank" links carry rel="noopener noreferrer"
  • Internal anchor links scroll smoothly; target offset accounts for sticky header
  • mailto: and tel: links open the user's email / dialer
  • Visited links are visually distinguishable
  • Visible focus ring on Tab; never globally suppressed
  • Automated 404 / broken-link checker runs in CI; build fails on dead links
  • Image links: entire image clickable; alt text describes destination

File flows are silent killers.

They look fine in dev — small test files, fast network — and break in production. Type, size, progress, and errors all need to be designed in from the start.

Bad — Will be rejected
No file-type filter, no size limit, no progress.
After clicking — page appears frozen
uploading_huge_file_500MB.exe
No progress bar. No cancel. User clicks Upload again.
Upload failed.
  • Server accepts disallowed file types (client-only check)
  • No size limit; users upload 500 MB files and break the disk
  • No progress indicator — user thinks the page is frozen
  • Cancel button missing or non-functional during upload
  • Generic upload errors with no reason given
Good — Passes review

  • Allowed file types enforced on both client and server
  • Size limit enforced; clear error states the actual limit
  • Progress bar with percentage during upload
  • Cancel button stops the upload and resets the input
  • Download filenames are descriptive (Invoice_2026-05.pdf)
  • Specific upload errors: type, size, password-protected, etc.
